## 3. Successes & Strengths

- **Architectural Isolation (Pure Resolver):** The creation of a fully decoupled combat resolver boundary (`module/combat/`) free of Foundry-specific globals (`game`, `ui`, `ChatMessage`) is a major victory. It guarantees that game mechanics can evolve independently of the VTT UI.
- **Foundry-Free Testing (Fixture Harness):** The new offline test suite runs instantly under Node.js. Developers can assert combat logic without running a local Foundry server, reducing verification loop times from minutes to seconds.
- **State Discipline (State Planner):** Enforcing that all combat outcomes return a list of planned updates instead of mutating documents in-place prevents state/chat divergence, a common source of bugs in Foundry systems.

---

## 4. Challenges & Growth Areas

- **Plain Data Cleanliness:** Code reviews for Story 1.3 and 1.4 revealed that serializing complex modifier options or live document instances can throw errors or lose data (JSON-unsafe payloads).
- **Growth Area:** We must enforce strict plain-data boundaries in our adapter layer (`CyberpunkItem`) before sending objects into the pure resolver.

---

## 5. Key Insights & Lessons Learned

1. **Foundry API Decoupling is a High-Yield Investment:** Pure logic is easier to write, refactor, and reason about.
2. **Fixture-First Development Works:** Authoring JSON fixtures containing expected outcomes forces clarity on rule contracts before writing implementation code.
3. **Defense Against Churn:** Sticking to Plain JS ES Modules without introducing complex NPM workflows kept the package lightweight and brownfield-friendly.
