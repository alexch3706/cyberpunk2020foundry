| D-1 | 2026-05-24 | Create a brownfield PRD focused on CP2020 core combat mechanical fidelity, not a full product relaunch. | The investigation found correctness blockers in mechanics resolution, not a need to replace the Foundry system stack. | Captured in PRD |
| D-2 | 2026-05-24 | Use the static audit as the primary input for MVP scope. | It contains evidence-graded P0/P1/P2 findings with code and corebook page references. | Captured in PRD |

| D-4 | 2026-05-24 | Treat the current PRD as draft until architecture review. | The open scope assumptions are now confirmed, but downstream architecture still needs to validate feasibility and sequencing. | Captured in PRD |
| D-5 | 2026-05-24 | Include staged armor penetration in Corebook Fidelity Mode and make it explicitly toggleable. | It is part of corebook-faithful damage resolution, but tables may need an off switch for compatibility or referee preference. | Captured in PRD |
| D-6 | 2026-05-24 | Use preview/confirm before applying damage to Target Actor in the first release. | This protects games from incorrect targeting or resolver mistakes while the mechanics layer is new. | Captured in PRD |
| D-7 | 2026-05-24 | Fix fast-draw initiative semantics in MVP; defer party initiative. | Fast draw is a common actor-level combat action; party initiative can remain manual/documented without blocking core combat resolver work. | Captured in PRD |
| D-8 | 2026-05-24 | For pack conformance, normalize source/core labels and add core vs extended visibility; do not physically split packs in MVP. | This creates a conformance boundary without making pack migration the dominant effort. | Captured in PRD |
| D-9 | 2026-05-24 | Cover is manual input in MVP, not scene/map object automation. | Map-aware cover would expand scope into geometry and line-of-sight automation. | Captured in PRD |
| D-10 | 2026-05-24 | MVP should support all corebook martial arts styles and key techniques, but data modeling and table population can be separate stories. | Partial style support would keep martial arts fidelity incomplete. | Captured in PRD |

## Resolved Scope Decisions

| ID | Question | Default Assumption | Impact |
| --- | --- | --- | --- |
| OQ-1 | Should staged armor penetration be required in MVP or behind a setting? | Resolved: include in Corebook Fidelity Mode, explicitly toggleable. | Affects armor resolver and item state updates. |
| OQ-2 | Should pack data be corebook-only in MVP, or can non-core sources remain visible with attribution? | Resolved: label/filter core vs extended; do not split packs in MVP. | Affects data work and release definition. |
| OQ-3 | Should the MVP automate actor damage updates, or only produce a trusted resolution card for manual application? | Resolved: preview/confirm before applying damage in first release. | Affects UX and Foundry permission handling. |
| OQ-4 | Should party initiative and fast-draw initiative be included in MVP? | Resolved: fast draw in MVP; party initiative deferred. | Affects combat workflow scope. |
| OQ-5 | How should cover work in MVP? | Resolved: manual cover input only. | Avoids scene geometry scope expansion. |
| OQ-6 | How complete should martial arts style support be? | Resolved: all corebook styles in MVP; data model and table population may be separate stories. | Affects martial data scope and acceptance criteria. |
