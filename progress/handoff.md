# Handoff Context: Codebase vs Corebook Comparison

## Цель следующей сессии
Сравнение текущей кодовой базы (Foundry VTT System для Cyberpunk 2020) с оригинальными правилами (Corebook), описанными в `docs/`. Задача — провести аудит соответствия механик (Combat Resolver) правилам из Corebook и задокументировать расхождения, реализованные фичи и пробелы.

## Контекст проекта
- **Проект**: Система Cyberpunk 2020 для Foundry VTT.
- **Текущий статус**: Завершена фаза `4-implementation` для Эпиков 1–6. Базовый MVP готов.
- **Главный паттерн архитектуры**: Вся логика вынесена в `module/combat/` (Combat Resolver) — это чистые JS-модули без привязки к Foundry API. 
- **Точка входа**: `CyberpunkItem` (в `module/item/item.js`) делегирует вызовы в resolver.
- **Управление состоянием**: Изменения брони, здоровья, патронов проходят через State Planner -> Preview/Confirm диалог -> коммит через API Foundry. `CombatOutcome` является единым источником правды.
- **Особенности механик**: 
  - Реализованы стрельба одиночными, очередями, suppressive fire, локации попадания, staged penetration, AP, BTM, спасброски, ближний бой, стили боевых искусств (Martial Arts).
  - Настройки системы включают `corebookFidelityMode` и `stagedPenetration`.

## Что нужно сделать агенту в следующей сессии:
1. Изучить папку `docs/`, где лежат материалы по Corebook (например, markdown версии правил).
2. Сравнить их с логикой в `module/combat/` (включая `attack-resolver.js`, `damage-resolver.js`, `armor-resolver.js` и т.д.).
3. Составить аудит-отчет или обновить документацию в `docs/`, отметив, насколько точно текущий код следует правилам настольной игры, где сделаны упрощения для Foundry, а где функционал еще предстоит реализовать.
